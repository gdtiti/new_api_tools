## 方案摘要

- 推荐方案:
  - 方案 A：基于当前中间件架构做“分析增强 + 管理增强”分层落地，而不是直接把 `new-api-gdtiti` 的主系统运行时逻辑平移过来。
- 推荐原因:
  - 当前项目已有稳定的日志聚合、缓存和后台 UI，适合承载渠道/订阅分析。
  - 当前项目不在主 relay 调度链路内，无法可靠承担“真实调度限流”；因此需要把“渠道并发”明确收敛为分析能力。
  - 临时账号最适合采用“主库标准实体 + sidecar 元数据”的中间件模式，能减少对主库 schema 的侵入。
  - 参考项目的 3 份已验收任务包可直接提供字段、调用链和运营语义，但当前项目必须按自身边界重组。

## 备选方案对比

### 方案 A

- 做法:
  - 渠道侧：
    - 在 `backend-py` 新增渠道分析服务与路由，复用 `logs`、`channels`、现有 dashboard 缓存/预热模式；
    - 输出渠道健康分、请求量、错误率、平均响应时间、最近测试时间、估算并发峰值等指标；
    - 前端新增渠道分析页或 Dashboard 深层卡片/抽屉。
  - 订阅侧：
    - 新增订阅分析服务与路由，直接读主库订阅表；
    - 若表/字段不存在，则返回显式“当前库未接入订阅能力”。
  - 临时账号侧：
    - 通过主库创建/管理普通 `users`、`tokens`；
    - 通过本地 `local_storage` 增加 `temporary_accounts` 一类 sidecar 表记录临时账号元数据、默认 token、创建人、备注与生命周期状态；
    - 前端新增临时账号管理页。
- 收益:
  - 与当前项目架构边界一致，可逐块落地。
  - 主库兼容风险较低，便于支持“标准 NewAPI”与“gdtiti 变体”两种环境。
  - 频道/订阅分析可以大量复用现有缓存与趋势统计能力。
- 代价:
  - 不能 1:1 复制 `new-api-gdtiti` 的运行时控制语义。
  - 需要新增本地 sidecar 元数据表和同步约束。
- 风险:
  - 估算并发口径如果表达不清，容易被误解为实时精确值。
  - 订阅表结构若和参考项目不一致，需要再做兼容层。

### 方案 B

- 做法:
  - 直接按 `new-api-gdtiti` 的模型和主库 schema 在当前项目中补全运行时能力与临时账号字段，尽量复刻主系统实现。
- 收益:
  - 与参考项目语义最接近。
- 代价:
  - 需要改动主库 schema 或强依赖目标实例已经是 `gdtiti` 变体。
  - 会削弱当前项目“独立中间件/零侵入”的定位。
- 风险:
  - 一旦接入的不是 `gdtiti` 数据库，功能将大面积不可用。
  - 当前项目仍不在 relay inline 路径，复制“并发限制影响调度”并不能真正生效。

## 锁定设计

- 入口模块:
  - 渠道分析：
    - `backend-py/app/dashboard_service.py`
    - 新增 `backend-py/app/channel_insights_service.py`
    - 新增 `backend-py/app/channel_insights_routes.py`
    - `frontend/src/components/Dashboard.tsx` 或新增 `frontend/src/components/ChannelInsights.tsx`
  - 订阅分析：
    - 新增 `backend-py/app/subscription_analytics_service.py`
    - 新增 `backend-py/app/subscription_analytics_routes.py`
    - 新增前端订阅分析组件
  - 临时账号管理：
    - 新增 `backend-py/app/temporary_account_service.py`
    - 新增 `backend-py/app/temporary_account_routes.py`
    - `backend-py/app/local_storage.py` 扩展 sidecar 表
    - 新增前端临时账号组件
- 关键调用链:
  - 渠道健康度：
    1. 前端请求渠道分析 API；
    2. 服务层聚合 `channels` 当前状态 + `logs` 时间窗口聚合；
    3. 输出健康度分层、异常渠道、趋势与 Top 渠道。
  - 渠道并发估算：
    1. 读取 `logs.channel_id / created_at / response_time / type`；
    2. 在固定时间桶内估算请求重叠；
    3. 输出峰值并发、P95 并发、热点渠道排序。
  - 订阅分析：
    1. 校验订阅相关表是否存在；
    2. 存在则读取套餐、用户订阅、活跃状态与使用量；
    3. 不存在则返回显式 unsupported 状态。
  - 临时账号管理：
    1. 管理端在中间件侧提交创建/管理请求；
    2. 服务层写主库 `users` / `tokens`；
    3. 同步写本地 sidecar 元数据；
    4. 前端使用 sidecar + 主库联合数据进行展示与管理。
- 数据流:
  - 渠道分析主数据来自主库：
    - `channels`
    - `logs`
  - 订阅分析主数据来自主库：
    - 订阅计划表
    - 用户订阅表
    - 可能的套餐使用/扣费事实表
  - 临时账号扩展数据来自：
    - 主库 `users` / `tokens`
    - 本地 sidecar `temporary_accounts`、操作审计表
- 错误处理:
  - 订阅相关表缺失：返回显式“不支持/依赖未满足”。
  - 渠道并发估算依赖字段缺失：返回显式“不支持当前日志结构”。
  - 临时账号创建主库成功、sidecar 失败时：必须显式回滚或标记不一致，不能静默吞错。
- 配置变更:
  - 可能新增本地分析缓存 TTL。
  - 可能新增临时账号默认策略配置（默认 token 名称、默认过期、默认配额策略）。

    - `20260327-233848-订阅套餐兑换码与临时账号开通`
- 实现事实源:
  - 当前项目 `dashboard_routes.py` 仅有 `/api/dashboard/channels` 基础状态接口。
  - 当前项目已广泛使用 `logs` 进行聚合分析，且索引中明确存在 `logs.channel_id`。
  - 当前项目无订阅分析模块、无临时账号模块。
  - 当前项目 wiki 项目概览与真实仓库架构不一致。
- need_communication:
  - yes
- 主要差异:
  - 参考项目是主系统，能直接改 relay 调度与主库 schema；当前项目是中间件，不能无代价复刻这一路径。
  - 参考项目的“渠道并发限制”是控制能力；当前项目更适合做“并发跟踪分析”。
  - 参考项目的临时账号依赖主系统实现；当前项目更适合通过 sidecar 元数据承载“临时账号身份”。
- 待沟通项:

  - 目标环境是否保证存在 `gdtiti` 订阅表；
  - 渠道分析页是否要做成独立 tab。
