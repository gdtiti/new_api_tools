## 问题定义

- 当前问题:
  - 当前项目 `new_api_tools` 已有 `dashboard`、`topups`、`redemptions`、`user_management`、`model_status` 等中间件能力，但仍停留在“平台总览 + 用户/模型分析”层，缺少面向渠道、订阅和临时账号的增强模块。
  - 渠道侧当前只有 `/api/dashboard/channels` 的基础状态概览，数据仅来自 `channels.status / balance / used_quota / response_time / test_time`，没有健康趋势、失败原因、渠道级明细分析，也没有并发分析视图。
  - 当前项目虽然直接读取 `logs` 表，且已知 `logs` 存在 `channel_id / created_at / type`，仓库中也多处使用 `response_time` 指标，但尚未形成“按渠道聚合的请求量、错误率、响应时延、估算并发峰值”的分析能力。
  - 当前项目没有订阅套餐统计与订阅使用情况模块；从路由注册、前端标签和 README 可见，现有公开能力不包含 `/api/subscription/*` 或后台订阅统计页。
  - 当前项目没有临时账号管理；虽然已有封禁/解禁用户和 token 级管理经验，但没有“临时账号识别、创建、默认 token、生命周期管理”的专门数据与界面。
  - 参考项目 `J:\\_Dev\\_me\\new-api-gdtiti` 已经分别落地了：
    - `频道多BaseURL自动禁用与健康检查`
    - `渠道并发数限制影响调度`
    - `订阅套餐兑换码与临时账号开通`
    证明这些能力是实际需求，不是预设问题。
- 影响对象:
  - 管理员 / 运维：需要定位坏渠道、观察热点渠道压力、查看套餐使用情况、集中管理临时账号。
  - 业务分析人员：需要看到套餐的在用明细、使用趋势、活跃套餐和渠道健康状态，而不是只看全局概览。
  - 当前项目后端：`backend-py/app/dashboard_*`、`log_analytics_service.py`、`user_management_*`、`local_storage.py` 需要扩展为新的分析与管理能力承载层。
  - 当前项目前端：`frontend/src/components/Dashboard.tsx`、现有 tab 结构，以及潜在的新渠道/订阅/临时账号页面会被影响。
- 触发背景:
  - 用户明确要求“对标 `new-api-gdtiti` 增强本项目”，目标聚焦在四条主线：渠道健康度、渠道并发数跟踪分析、订阅套餐分析统计/使用情况/在用套餐列表、临时账号管理。
  - 当前项目本质是“独立认证的中间件 + 直接访问 NewAPI 主数据库 + 本地 SQLite/Redis 缓存/审计”，适合承载后台分析与管理增强，但并不在主 relay 调度链路内。
  - 参考项目已经证明：
    - 渠道健康度与并发限制在主系统中是值得做的；
    - 订阅与临时账号能力在业务上已经被真实使用；
    - 当前项目可以借这些事实源做“分析增强版”或“管理侧增强版”，但不能照搬主系统运行时假设。
- 为什么现在处理:
  - 当前项目已经具备可复用的日志分析、缓存、用户管理和管理后台框架，继续扩展的边际成本低于从零新建。
  - 若继续缺失渠道/订阅/临时账号增强，当前项目就只能提供“平台总览”，无法对标用户已经在 `new-api-gdtiti` 中使用的更深层运营分析能力。
  - 趁对标对象和近期任务包都在本地可读、实现事实明确，现在完成 discovery 最容易形成可靠的增强计划。

## 目标与成功标准

- 目标 1:
  - 为当前项目补齐“渠道健康度跟踪”能力，至少覆盖渠道可用状态、近期请求量、成功/失败分布、响应时延和可解释的健康分层。
- 目标 2:
  - 为当前项目补齐“渠道并发数跟踪分析”能力，但口径明确为**基于日志和响应时长的分析/估算**，不是对主系统调度器的硬限流。
- 目标 3:
  - 为当前项目补齐“订阅套餐统计与使用情况分析”，包括套餐总览、使用趋势、活跃套餐数、在用套餐列表明细。
- 目标 4:
  - 为当前项目设计并落地“临时账号管理”入口，支持最小必要的创建、查看、禁用/恢复和默认 token 管理。
- 目标 5:
  - 在增强过程中保持当前项目的中间件定位：优先复用主数据库现有事实，必要时把项目专属元数据落在本地 sidecar 存储，而不是无证据地改造主系统运行时链路。

## 范围边界

### 本次必须完成

- 渠道分析：
  - 新增渠道健康度分析 API 与页面/区块；
  - 聚合 `channels` 与 `logs`，输出请求量、错误率、平均响应时间、最近测试时间、健康分层；
  - 明确并实现“估算并发”口径，至少提供窗口内峰值/分位值分析。
- 订阅分析：
  - 新增订阅套餐统计与在用套餐明细分析能力；
  - 若主数据库缺少订阅相关表或字段，返回显式“依赖未满足/当前库不支持”，不能静默伪造数据。
- 临时账号管理：
  - 形成可实施的数据设计与接口设计；
  - 推荐以“主库标准用户/token + 本地 sidecar 临时账号元数据”的方式实现，以减少对主库 schema 的侵入。
- 任务文档产出：
  - 明确当前项目与参考项目的能力映射；
  - 明确哪些能力可直接复用，哪些能力只能做分析版，哪些能力需要新元数据表。

### 本次明确不做

- 不把当前中间件改造成主系统 relay 入口，不在本项目里实现 `new-api-gdtiti` 那种“调度层真实并发限制”。
- 不在没有证据的情况下假设所有接入实例都已经具备 `new-api-gdtiti` 的订阅/临时账号 schema 变更。
- 不做静默回退：依赖缺失时必须显式报出“不支持/待接入”，不能返回假统计。
- 不在 discovery 阶段直接进入主工作区编码，也不跳过 worktree 门禁。

## 验收真相陈述

- [REQ-001] 当前项目的增强方案必须明确区分“可直接在中间件实现的分析能力”和“只有主系统 inline 才能做到的运行时控制能力”。
- [REQ-002] 渠道健康度方案必须能说明将复用哪些现有表/字段/模块，且至少包含状态、请求量、错误率、响应时延四类指标。
- [REQ-003] 渠道并发方案必须明确其为“估算分析”而非“硬限制”，并给出估算所依赖的日志字段与算法入口。
- [REQ-004] 订阅分析方案必须覆盖总览统计、使用情况分析、在用套餐列表明细三类输出，并写清主数据库依赖。
- [REQ-005] 临时账号管理方案必须给出数据归属策略：哪些数据写主库、哪些数据写本地 sidecar，以及相应风险。
- [REQ-006] Discovery 结束时必须能输出分阶段实施计划、影响模块、验证方式和至少一项关键待确认问题。

## 约束与依赖

- 技术约束:
  - 当前项目真实代码形态以仓库事实为准：`backend-py` FastAPI + `frontend` React/Vite；`.devone/wiki/projects/README.md` 中的 “Go + nginx + redis” 描述已与当前仓库不一致，不能作为实现事实源。
  - 当前项目是中间件，直接访问主数据库并维护本地 SQLite/Redis 缓存；它不在主系统请求调度链路中。
  - 当前项目已存在日志分析、缓存预热、用户管理、本地 sidecar 存储能力，应优先复用：`dashboard_service.py`、`log_analytics_service.py`、`local_storage.py`、`user_management_service.py`。
  - 日志分析必须考虑大表场景，沿用现有“大系统增量缓存/预热”设计，避免直接上高代价全表查询。
- 业务约束:
  - “并发数跟踪分析”与“渠道并发限制”不是一回事；本轮默认只做前者。
  - 临时账号管理需要可解释、可回收，不能仅靠普通用户列表打标签人工辨认。
  - 订阅分析应以真实主库事实为准；若表不存在，系统必须显式提示依赖缺失。
- 外部依赖:
  - 参考事实源：`J:\\_Dev\\_me\\new-api-gdtiti` 中 3 个已落地任务包。
  - 主库依赖：`channels`、`logs`、以及可能存在的订阅相关表。
  - 本地依赖：当前项目自带的 `local_storage` / 缓存 / 审计表能力。

## 风险与待确认项

- 风险:
  - 当前项目不在 relay 链路内，若直接承诺“实时精确并发”，会超出架构可兑现范围。
  - 订阅功能在当前项目中没有已实现模块，若目标实例并非 `new-api-gdtiti` 变体，可能不存在所需表/字段。
  - 若临时账号直接改主库 schema，会破坏当前项目 README 中“零侵入 / 直连主库”的定位。
  - 当前项目 wiki 中已有过期项目概览，若不先纠偏，后续 execution 易基于错误架构假设展开。
- 待确认:
  - 待确认 1：临时账号管理是否接受“sidecar 元数据 + 主库标准用户/token”的中间件方案，而不是要求完全复刻 `new-api-gdtiti` 的主库 schema 改动。
  - 待确认 2：订阅分析的目标实例是否就是 `new-api-gdtiti` 数据库；若不是，需要先核对表结构兼容性。
  - 默认假设 A：本轮渠道并发仅做分析估算，估算依据为 `logs.channel_id + logs.created_at + 响应时长字段`。
  - 默认假设 B：订阅分析优先做后台统计与明细，不在第一阶段复刻完整门户订阅中心。
  - 任务形态判断：`epic`。原因是同时覆盖渠道分析、订阅分析、临时账号管理三条能力链，且每条链都跨后端、前端、数据与验证。
